HTTP:Web基础——URL与资源

URL与资源

浏览Internet资源

URI由URL和URN构成。URL Vs. URN URL是通过描述资源的位置来标识资源的，而URN则是通过名字来识别资源的，与它们当前所处的位置无关。URL提供了一种统一的资源命名方式，大多数URL都有同样的：“方案：//服务器位置/路径”结构。以下展示URL的三部分。

同时，URL也可通过HTTP外的其他协议来访问资源，它们可以指向Internet上的任意资源。

URL的语法

方案

方案是规定如何访问制定资源的主要标识符。方案名与大小写无关。

主机与端口

要想在因特网上找到资源，应用程序要知道是哪台机器装载了资源，以及在哪台机器的什么地方可以找到对目标资源进行访问的服务器。URL的主机和端口组件提供了这两组信息。对下层使用了TCP协议的HTTP来说，默认端口号为80.

用户名和密码

很多服务器都要求输入用户名和密码才会允许用户访问数据。eg. FTP

路径

路径通常很像一个分级的文件系统路径。每个路径段都有自己的参数组件。

参数

为了向应用程序提供它们所需的输入参数，以便正确地与服务器进行交互，URL中有一个参数组件。HTTP URL 的路径组件可以分成若干路径段。每段都可以有自己的参数。

查询字符串

很多资源，eg.数据库服务，都是可以通过提问或进行查询来缩小所请求资源类型范围的。除了有些不合规则的字符需要特别处理外，对查询组件的格式没什么要求。按照常规网关都希望查询字符串以一系列“名/值”对的形式出现，名值对之间用“&”分隔。

片段

以便引用部分资源或资源的一个片段，URL支持使用frag组件来表示一个资源内部的片段。HTTP服务器通常只处理整个对象，而不是对象片段，客户端不能将片段传送给服务器。浏览器从服务器获得了整个资源后，会根据片段来显示你感兴趣的那部分资源。

注意： 片段是不会发送给服务器的，服务器会返回整个HTML页面。是由浏览器从指定片段处开始显示HTML页面的。

URL快捷方式

相对URL

URL有两种方式： 绝对的和相对的。绝对URL中包含所有访问资源所需的全部信息。相对URL是URL的一种便捷缩略记法，是不完整的，想从相对URL中获取访问资源所需的全部信息，就必须相对于另一个，被称为基础（base）的URL进行解析。方案、主机和其他一些组件可以从它们所属资源的基础URL中推导出来。处理URL的应用程序要能在相对和绝对URL之间进行转换。还需注意的是： 相对URL为保持一组资源的可移植性提供了一种便携方式。如果使用的是相对URL，就可以在搬移一组文档的同时，仍然保持链接的有效性，因为相对URL都是相对于新基础进行解释的。这样就可以实现在其他服务器上提供景象内容之类的功能了。

基础URL

可来自于：

  • 在资料中显式提供： eg. HTML文档中可能会包含一个定义了基础URL的HTML标记 BASE
  • 封装资源的基础URL： 如果在一个没有显式指定基础URL的资源中发现了一个相对URL，可以将它所属的资源的URL作为基础。
  • 没有基础URL：通常意味着你有一个绝对URL；但有时可能只是一个不完整或损坏了的URL。
解析相对引用

分解URL。只要将基础和相对URL划分成了组件，就可以通过下属将相对URL转换成其绝对模式，之后就可以用它来引用资源了。

自动扩展URL

有些浏览器会在用户提交URL之后，或在用户输入的时候尝试着自动扩展URL。“自动扩展”特性有以下两种方式：

  • 主机名扩展：只要有一些小提示，浏览器就可以在没有帮助的情况下，将你输入的户籍名扩展为完整的主机名。eg. yahoo–>www.yahoo.com。 但这可能会为其他一些HTTP应用程序带来问题，比如代理。
  • 历史扩展： 浏览器将以前用户访问过的URL历史存储起来，当用户输入时，就将输入与历史URL的前缀向匹配，并提供一些完整的选项供你选择。注意： 与代理共同使用时，URL自动扩展的行为可能会有所不同。

字符

URL字符主要考虑到三点：

  • 可移植，安全传输。它要统一地命名Internet上所有的资源，也就意味着要通过各种不同的协议传送这些资源，安全传输意味着URL的传输不能丢失信息。有些协议使用的传输方法会除去一些特定的字符。为了避开这些问题，URL只能使用一些相对较小的、通用的安全字母表中的字符。
  • 可供人类阅读，因此即使不可见、不可打印的字符能够穿过邮件程序，从而成为可移植的，也不能在URL中使用。
  • 完整。有时人们可能会希望URL中包含除通用的安全字母表之外的二进制数据或字符。因此需要有种转椅机制，能够将不安全的字符编码为安全字符，再进行传输。

URL字符集

US-ASCII字符集可以执行很好，但完整性欠佳。所以URL的设计者就将转义序列继承了进去，通过转义序列，就可以用US-ASCII字符集的有限子集对任意字符值或数据进行编码，这样就实现了可移植性和完整性。

编码机制

用来在URL中表示各种不安全的字符，这种编码机制就是通过“转义”表示法来表示不安全的字符，这种转义表示法包含一个百分号%，后面跟着两个表示字符ASCII码的十六进制数。

字符限制

  • 有着特殊含义。
  • 不在定义的US-ASCII可打印字符集中。
  • 与某些因特网网关和协议产生混淆。

说明

应用程序要按照一定规范工作。客户端应用程序在向其他应用程序发送任意URL之前，最好把所有不安全或受限字符都进行转换。只要对所有不安全字符都进行了编码，这个URL就是可在各应用程序之间共享的规范格式；也就无需操心其他应用程序会被字符的任何含义所迷惑。
最适合判断是否需要对字符进行编码的程序就是从用户处获取URL的远端应用程序。

总结

URL是一种强有力的工具。但URL并不完美。它们表示的是实际的地址，而不是准确的名字。这种方案的确定在于如果资源被移走了，URL就不再有效。那是，它就无法对对象进行定位了。
因此IETE已经对URN的新标准做了一段时间的研究了。PURL就是用URL来实现URN功能的一个例子。基本思想是在搜索资源的过程中引入另一个中间层，通过一个中间资源定位符服务器对资源的实际URL进行等级和跟踪。客户端可以向定位符请求一个永久URL， 定位符可以以一个资源作为相应，将客户端重定向到资源当前实际的URL上去。
